# BIP110 (Reduced Data Temporary Softfork) Plan for forseti

> **STATUS (2026-07-10): DESIGN ONLY — NOT IMPLEMENTED.** This documents how we
> *would* add BIP110 if we chose to, behind an opt-in flag that is **off by
> default**. Nothing here ships until explicitly greenlit. BIP110 is a
> contentious, non-activated proposal; forseti's default behavior must not
> change.

## What BIP110 is

BIP110 ("Reduced Data Temporary Softfork", Dathon Ohm) is a **consensus soft
fork** that, for a ~1-year window, tightens several script/output rules to cap
arbitrary on-chain data. It applies **only to UTXOs created at or after the
activation height** — pre-existing coins are grandfathered, so **no funds can be
frozen**. Proposed deployment is a modified BIP9 (55% threshold, ~Sept 2026).

Source: <https://github.com/bitcoin/bips/blob/master/bip-0110.mediawiki>

### Reference implementation (port from this, don't reinvent)

There is a working reference implementation in Bitcoin Knots (C++):

- **`dathonohm/bitcoin`** — the BIP author's fork, tag
  `v29.3.knots20260210+bip110-v0.3` (also RC1/RC3 tags). This is the canonical
  BIP110 code.
- Now merged into **Bitcoin Knots** itself (`bitcoinknots/bitcoin`, release
  `v29.3.knots20260508`) — which, notably, **requires explicit operator
  confirmation to enable** (validating our off-by-default + explicit-opt-in
  stance) and **defaults `datacarriersize` to 83** (exactly forseti's default —
  no change needed there).
- **`feature_rdts.py`** in that fork's functional test suite exercises **all seven
  rules, their boundaries, and the rule-2 script-type exemptions** (BIP16
  redeemScripts, witness scripts, Tapleaf scripts). These test scenarios are
  language-agnostic (specific tx/block constructions) and are the **highest-value
  thing to port** — they become forseti's consensus test vectors directly, even
  though the C++ enforcement code itself must be re-expressed in Odin.

Porting strategy: read Knots' consensus diff to get the exact predicates and
boundary conditions right (especially the UTXO-age gate and the window expiry),
then translate `feature_rdts.py` into forseti's test suite as the acceptance bar.

**Naming note:** the spec is BIP110, but the Knots code names the deployment
`reduced_data` / `DEPLOYMENT_REDUCED_DATA`, and Knots' own test/release notes
reference **BIP444** — the numbering appears to have evolved. Track by the
`reduced_data` deployment, not the BIP number.

### Concrete constants & parameters (gathered from the reference impl + spec)

Exact values (from Knots `feature_rdts.py` / constants):
- `MAX_OUTPUT_SCRIPT_SIZE = 34` (rule 1; new non-OP_RETURN scriptPubKeys)
- `MAX_OUTPUT_DATA_SIZE = 83` (rule 1; OP_RETURN outputs)
- `MAX_SCRIPT_ELEMENT_SIZE_REDUCED = 256` (rule 2; pushes / witness elements)
- `TAPROOT_CONTROL_MAX_SIZE_REDUCED = 257` (rule 5; = 33 + 32*7, i.e. ≤7 merkle
  nodes / ~128 leaves)

Reject reasons (map to forseti's error enum): `bad-txns-vout-script-toolarge`
(create-side output size); spend-side rules surface as
`non-mandatory-script-verify-flag` (i.e. a `script.Script_Error` behind the
reduced-data verify flag).

**Rule-2 exemptions are STRUCTURAL, not special-cased** — this is the key
implementation insight from the reference test: the redeemScript, witness script,
and Tapleaf script are **popped off the stack BEFORE** the element-size check runs
(in `ExecuteWitnessScript`), so they are *implicitly* exempt. PUSHDATAs *inside*
those scripts are still limited. So forseti's port should run the 256-byte
element check on the remaining stack after popping the script — the exemption
falls out of ordering, no explicit carve-out list needed. (`feature_rdts.py`
verifies this with an 11-of-15 P2WSH multisig ~513-byte witness script and a
>256-byte tapleaf built from repeated `<data> OP_DROP`.)

Deployment / activation (spec — modified BIP9, block-height versionbits):
- deployment name `reduced_data`, **bit 4**, **55% threshold** (1109/2016, not the
  usual 95%)
- starttime ~2025-12-01 (1764547200); max activation height **965664** (~Sep 2026);
  mandatory signaling 961632–963647; lock-in 963648
- **`active_duration = 52416` blocks (~1 yr)**: after activation, state goes
  ACTIVE → EXPIRED; rules stop at `activation_height + 52416`.

**Activation vs grandfathering are two separate gates — this is the crux:**
1. *Activation* (versionbits, **block height**) decides the activation height H and
   the expiry H+52416. Standard soft-fork machinery; forseti already has
   height-gated flags (`get_script_flags`).
2. *Grandfathering* (**per-input, UTXO creation height**) — once active, the
   spend-side rules apply to an input **only if the coin it spends was created at
   height ≥ H** (and, given expiry, presumably `< H+52416` for created coins — the
   exact window semantics for coins created inside the window vs spent after it
   must be read from the spec/code, not assumed). This per-input gate is the real
   architectural work (see "Spend-side" below) and is **NOT covered by
   `feature_rdts.py`**, which runs `-vbparams=reduced_data:-1:...` (ALWAYS_ACTIVE
   from block 0, so no pre-activation UTXOs exist and grandfathering is vacuous).
   A forseti port must ADD age-gate boundary tests on top of the ported suite.

### The seven rules, split by where they apply

BIP110's rules fall into two very different buckets, and the split drives the
whole implementation:

**A. Create-side (gate on the *block's* height ≥ activation).** These constrain
*outputs being created*, so they simply apply to any output in a block at/after
the activation height:

1. New scriptPubKeys > 34 bytes are invalid — **except** `OP_RETURN`, allowed up
   to 83 bytes. (Note: 83 is exactly forseti's `--datacarriersize` default, which
   we deliberately keep — see docs/usage; BIP110 would make that a *consensus*
   bound for new outputs, not just relay policy.)

**B. Spend-side (gate on the *spent UTXO's* creation height ≥ activation).** These
constrain *how a coin is spent*, and — critically — only for coins **created**
at/after activation. This UTXO-age gate **is** the grandfathering mechanism:

2. `OP_PUSHDATA` payloads and witness stack elements ≤ 256 bytes — except BIP16
   redeemScripts, witness scripts, and Tapleaf scripts.
3. Spending an undefined witness version or undefined Tapleaf version is invalid.
4. A Taproot annex in the witness stack is invalid.
5. Taproot control blocks ≤ 257 bytes (~128 script leaves).
6. `OP_SUCCESSx` opcodes are prohibited anywhere in Tapscript.
7. `OP_IF` / `OP_NOTIF` cannot be executed in Tapscript.

The bucket-B rules re-tighten relaxations that BIP342 deliberately left open for
future upgrades (undefined versions, `OP_SUCCESS`, annex) — worth flagging as a
tension with Taproot's upgrade path.

## Goal

Add BIP110 support behind a `--bip110=off|track|enforce` CLI flag (default
**off**), mirroring the existing `--drivechain=off|track|enforce` design
(docs/plans/drivechain.md, `drivechain.Mode`):

- **`off`** (default): byte-for-byte today's behavior. No hooks, zero cost.
- **`track`**: check every rule and **log** violations, but reject nothing —
  zero-consensus-risk observation. Lets a node operator see how much traffic
  *would* be affected without risking a split.
- **`enforce`**: additionally reject blocks/txs that violate the rules.

## "Is opt-in even possible?" — yes, with a sharp caveat

Technically yes: forseti already does exactly this for drivechain (`enforce`
mode applies consensus rules the rest of the network doesn't). But the semantics
matter, and they are **not symmetric with a normal config toggle**:

- **`off` / `track` are always safe.** A soft fork is backward-compatible: if
  BIP110 *does* activate network-wide, a non-enforcing forseti node still follows
  the same chain (it accepts the stricter blocks miners produce). So a default
  node needs no changes to stay in consensus whether or not BIP110 activates.

- **`enforce` is safe ONLY once BIP110 is actually active on the network.** The
  self-fork risk does **not** exist immediately on enabling the flag, and does not
  fork "right away":
  - **Below `bip110_height`, enforce is a complete no-op** — the rules don't apply
    (grandfathering), nothing is rejected.
  - **From `bip110_height` onward**, enforce rejects a block only if that block
    actually contains a BIP110-violating transaction.
  - **The fork materializes at the first block where enforce rejects something the
    network accepted** — the first post-activation block carrying a
    violating-but-network-valid tx. That may be at the activation height, many
    blocks later, or never.
  - **And only if forseti enforces while the majority does not.** If BIP110
    genuinely activates network-wide, miners won't include violating txs in
    post-activation blocks, so enforce never rejects and stays in consensus. The
    self-fork is strictly the "I enforce a rule the majority doesn't" case.

  So: the risk *opens* at `bip110_height` and *materializes* at the first
  network-accepted violating block — identical in kind to the drivechain-`enforce`
  self-fork risk already documented in docs/bips.md and surfaced in `--help`.

**Conclusion:** opt-in is possible and off-by-default is the right stance.
`enforce` must ship with a loud startup warning (as drivechain does) and should
arguably refuse to enable unless the operator also sets an explicit
`--bip110height=<N>` acknowledging they know when they intend the rules to bind.
`track` is the genuinely useful mode for most operators.

## How it hooks into forseti

Two insertion points, matching the create/spend split above.

### Create-side (rule 1) — `consensus` / `chain`

Output size checks are pure and block-height-gated. Add them where block outputs
are already walked during connect. Precedent: `chain/chainstate.odin`
`_is_unspendable` already inspects every output's scriptPubKey. A sibling check:

```
// pseudo — in the output loop of connect_block / check_block, gated on height
if mode != .Off && block_height >= params.bip110_height {
    for out in tx.outputs {
        spk := out.script_pubkey
        is_opreturn := len(spk) >= 1 && spk[0] == 0x6a
        limit := is_opreturn ? 83 : 34
        if len(spk) > limit {
            if mode == .Enforce { return .Bip110_Oversized_Output }
            else /* Track */     { log.warnf("BIP110: oversized %s output ...") }
        }
    }
}
```

Gate on the **block** height because these rules constrain outputs *created* in
that block.

### Spend-side (rules 2–7) — `script.Verify_Flags`, per-input by prevout age

These are script-execution rules, so they belong in the interpreter behind a new
verify flag, exactly like `.Taproot`:

1. Add `.BIP110` to `script.Verify_Flags` (`script/flags.odin`, alongside
   `.Taproot` at line ~47).
2. Enforce rules 2–7 in `script/interpreter.odin` (and the Taproot/Tapscript path
   for 3–7) when `.BIP110 in flags`. Rule 2's push-size cap sits in the pushdata
   handling with the redeemScript/witnessScript/Tapleaf carve-outs; rules 3–7 sit
   in the witness-program / control-block / Tapscript-opcode handling.
3. **The key architectural change:** the `.BIP110` flag is **per-input**, keyed on
   the *spent coin's* creation height, not the block height. Today every input in
   a block shares one flag set from `get_script_flags(block_height, params)`
   (`consensus/validation.odin:310`). For BIP110 spend rules, each input's flags
   depend on the age of the coin it spends:

   ```
   // per input, when building the Check_Task (chain/parallel.odin: Check_Task.flags):
   flags := base_flags                 // = get_script_flags(block_height, params)
   if mode != .Off && prevout_coin.height >= params.bip110_height {
       flags += {.BIP110}
   }
   ```

   The spent coin's creation height is already available at verification time:
   `Cache_Entry.coin` is a `storage.UTXO_Coin` with a `height: u32` field
   (`storage/utxodb.odin:7`), fetched for every input during connect. So no new
   data plumbing is needed — only that the flag be computed per-input instead of
   once per block. `chain/parallel.odin` already carries a per-task
   `flags: script.Verify_Flags` (line 20), so this is a clean insertion: set each
   task's flags from its input's prevout height.

   This per-input gate **is** BIP110's grandfathering: coins created before
   `bip110_height` never get `.BIP110`, so their spends are unaffected — funds
   cannot be frozen. Getting this boundary wrong (applying spend rules to old
   coins) would reject valid spends and split the chain, so it is safety-critical
   and must be the most heavily tested part.

`track` mode: run the same checks but downgrade a violation to a `log.warnf`
instead of a script/connect failure. Cleanest is a mode-aware wrapper so the
interpreter can report "would-fail" without aborting — or, simpler, run the
checks in a separate observation pass in `connect_block` and never set the flag
in `track`. (Interpreter-internal `track` is messier because a script failure
aborts execution; a pre-pass over the block's inputs/outputs is easier to reason
about and keeps `off`/`track` from touching the hot verify path.)

## Config & params

- `consensus/params.odin`: add `bip110_height: int` per network (default a
  sentinel like `max(int)` / 0 meaning "not set", since there is no real
  activation height baked in — the operator supplies it).
- `Bip110_Mode :: enum { Off, Track, Enforce }` (mirror `drivechain.Mode`).
- `main.odin`: `--bip110=off|track|enforce` (default `off`) + `--bip110height=<N>`;
  config-file support in the per-network section (mirror the drivechain flag
  wiring at `main.odin:942`). `enforce` without an explicit height → refuse to
  start (or warn hard), because a wrong height is a self-fork.
- Thread the mode into `Chain_State` (like `dc_mode`) so `connect_block` and the
  verification dispatch can read it.

## Activation

BIP110's real deployment is modified-BIP9 (55%, ~Sept 2026). For an **opt-in**
node we do **not** need to implement BIP9 signaling to enforce — enforcement is
flag+height driven, same as drivechain. Optionally we could *observe* BIP9
signaling for a BIP110 deployment (surface it in `getdeploymentinfo`) to help the
operator decide when it is safe to flip `track`→`enforce`, but that is a nicety,
not a requirement.

## Testing

**Port `feature_rdts.py` from the reference implementation as the acceptance
bar** — it already encodes all seven rules, boundaries, and rule-2 exemptions as
concrete tx/block scenarios. Translate each scenario into a forseti test; if they
all pass in `enforce` (and are all no-ops in `off`), the port is faithful. The
below is the same coverage, restated:

- **Create-side:** outputs of 34 / 35 bytes and OP_RETURN of 83 / 84 bytes at
  `bip110_height − 1` (allowed) vs `bip110_height` (rejected in enforce, logged in
  track, ignored in off).
- **Spend-side age gate (most important):** a coin created at `bip110_height − 1`
  spent with a would-violate witness (e.g. a 300-byte stack element) must be
  **accepted** (grandfathered); the same coin created at `bip110_height` must be
  **rejected** in enforce. Boundary tests at exactly the activation height.
- **Each rule 2–7** individually, with and without the carve-outs (redeemScript /
  witnessScript / Tapleaf push exemptions).
- **Mode matrix:** `off` never rejects/logs; `track` logs but never rejects (chain
  advances identically to `off`); `enforce` rejects.
- **Regression:** with `--bip110=off`, a full mainnet sync is byte-for-byte
  unchanged (no perf or behavior delta) — the `off` fast-path must be a true
  no-op, like `--drivechain=off`.

## Risks & open questions

- **Self-fork (enforce):** the headline risk — enforcing before/without network
  activation splits the node. Loud warnings + explicit height required.
- **Age-gate correctness is fund-safety-critical.** A bug that applies spend
  rules to pre-activation coins = rejected valid spends = frozen-looking funds +
  chain split. Fuzz the boundary.
- **Per-input flags** slightly change the verify dispatch (flags no longer
  uniform per block). Cheap — the prevout height is already fetched — but it
  touches the parallel-verification hot path; keep the `off` path branch-free.
- **Tapscript re-tightening vs future soft forks:** rules 3/4/6 forbid undefined
  witness/Tapleaf versions, the annex, and `OP_SUCCESS`. If a *real* future
  upgrade uses any of these, a node stuck in `enforce` past BIP110's 1-year expiry
  would wrongly reject it. The "temporary" expiry must itself be enforced (rules
  stop applying after the window) — another height/window to get right.
- **Contentious status:** BIP110 is not adopted; treat this as optional research
  tooling (`track` for measurement), not a correctness requirement.

See docs/plans/drivechain.md for the opt-in `off|track|enforce` pattern this
mirrors, and docs/bips.md for how self-fork risk is documented for operators.
